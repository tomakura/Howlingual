# リファクタリング概要

## 目的
- OCR以外のコードも含めて整理し、保守性を上げる。
- 既存の挙動は維持したまま、重複ロジックを減らす。
- マルチモニタ環境や高 DPI 環境でも安定してキャプチャ・表示・OCR が行えるようにする。

## 変更点
- ウィンドウ位置のクランプ処理を共通関数に集約。
- 重複していた`cfg`指定を整理。
- マルチモニタ対応のため、キャプチャ結果をモニタ（またはキャプチャ ID）単位で管理する`CapturedImages`用の`HashMap`構造を導入。
- キャプチャ処理全体をマルチモニタ前提のフローに整理し、各モニタごとの領域計算と画像取得を共通化。
- OCR 前処理（例: グレースケール化、二値化、トリミングなど）を追加し、OCR 用の画像パイプラインを専用関数群に切り出し。
- キャプチャ中の OCR をユーザー操作でキャンセルできるよう、`cancel_selection_ocr` 関数を追加。
- フロントエンドで`devicePixelRatio`を考慮したスケーリングを導入し、HiDPI 環境でもキャプチャオーバーレイや選択枠の位置ずれが起きないように修正。
- キャプチャオーバーレイ（選択枠・マスクなど）の描画ロジックを整理し、マルチモニタ＋スケーリング前提での座標変換・矩形計算を改善。

## 理由
- 位置計算のロジックを一箇所にまとめることで、将来の修正漏れを防ぐため。
- 無駄な重複指定を減らし可読性を上げるため。
- これまで不安定だったマルチモニタ環境でのキャプチャ処理を、モニタ単位の`CapturedImages`管理により一貫性のある設計にするため。
- OCR 前処理を明示的なパイプラインとして分離することで、精度向上やアルゴリズム差し替えを行いやすくするため。
- `cancel_selection_ocr` を導入することで、ユーザーがキャプチャ途中で操作を中断した際にも安全に処理を打ち切り、リソースリークや UI の中途半端な状態を防ぐため。
- `devicePixelRatio` を考慮したスケーリングとオーバーレイ改善により、高 DPI ディスプレイでも視覚的なズレやクリック位置の不一致が発生しないようにするため。
